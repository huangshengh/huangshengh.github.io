这个lab是做内存管理的。说实话最近我也在看内存管理这方面，但是感觉不是很懂。涉及到位图，内核和用户进程。不过最近的学习状态也不好，不多说了，来看实验吧。

内存管理呢需要两个部分，一个是物理内存分配器，我们用他来申请和释放以页为单位的空间并且我们还要维护一个数据结构表示哪个物理页是空闲的哪个是分配的，这个物理结构估计就是位图了。另一个部分是虚拟地址，我们需要建立页表来管理虚拟地址。

在开始之前呢我们要知道，我们是逐步的完成一个os，所以这次lab2是一个新的分支，但是之前lab1做的操作也要和lab2合并，这些都是在一个文件里面完成的。不知道之前lab1没有得到满分会不会影响。

lab2里面我们要看一下以下文件：

  • inc/memlayout.h
  • kern/pmap.c
  • kern/pmap.h
  • kern/kclock.h
  • kern/kclock.c

memlayout.h是虚拟地址空间的布局，kclock是关于cpu时序个内存大小的。

Memlayout.h pmap.h 要特别注意

首先我们来了解一下pc里面的内存情况。我们对于内存的管理是基于页大小来讲的，一个物理页是4K，管理页的结构是页表，每个页表管理1024个物理页，也就是一个页表管理的内存是4M。页表里面的1024个entry和物理页是一一对应的，每个entry是4个字节，其中前20位是物理页的起始地址，后面12位是属性。现在一般都是采用二级页表来管理，也就是有一个页目录表来管理页表，每个页目录项对应一个页表。

Part 2: Virtual Memory

####Exercise2

这个练习让我们看看页表转换和保护模式。

在x86下，虚拟地址是段选择子和段内偏移地址。线性地址是经过段内转换之后的虚拟地址，线性地址再经过页表转换形成物理地址。

####Exercise3：

第三个实验就是让我们用qemu的xp和gdb的x命令来看物理地址和对应虚拟地址上的内容。

主要是进入qemu的monitor模式。

在我们的操作系统中，只要我们经过启动之后。在程序里面见到的地址都是虚拟地址。像指针的值啊，这些指的都是虚拟地址，需要经过MMU（硬件）翻译成物理地址。

我们可以dereference一个虚拟地址，通过把它转换成一个指针。但是我们不能dereference一个物理地址，因为mmu翻译所有的内存引用，如果你强行把物理地址也给转换成指针可能会出错。

内核有时候需要直接读取和操作只知道物理地址的内存。比如为页表添加一个映射需要我们申请物理内存去存储一个页目录项并且初始化它。然而内核不能绕过虚拟地址翻译这一步所以我们不能直接的申请和加载物理地址。我们jos把从0开始的物理地址重新映射成从0xf0000000开始的虚拟地址就是帮助内核读写物理地址.KADDR（pa）就是把物理地址加上0xf0000000.PADDR(va)是把虚拟地址转换成物理地址的。

Reference counting

之后的lab中，我们的每个物理页可能同时的指向多个虚拟地址。PageInfo结构里面的pp_ref就是对于每个物理页的引用计数。只有当这个计数为0的时候这个页才能被释放。通常情况下，计数等于物理页在utop下出现在各个也表中的次数。因为utop是用户可以访问的虚拟地址的顶端。
