The ROM BIOS

​ 我们在lab目录启动两个终端，一个make qemu-gdb，一个是make gdb. make qemu-gdb 会在qemu模拟器执行第一条指令的时候停下，我们通过 make gdb 来看看这第一条指令到底是什么：

第一条指令是：
[f000:fff0] 0xffff0: ljmp $0xf000,$0xe05b
不想翻译了，我们直接看：

  1. The IBM PC starts executing at physical address 0x000ffff0, which is at the very top of the 64KB area reserved for the ROM BIOS.

  2. The PC starts executing with CS = 0xf000 and IP = 0xfff0.

  3. The first instruction to be executed is a jmp instruction, which jumps to the segmented address CS = 0xf000 and IP = 0xe05b。

    ​ 首先 我们要了解一个概念BIOS是什么，BIOS 全称：Base Input&Output System,就是基本输入输出系统。他是放在计算机的rom里面的，机器一启动就会去让BIOS启动，BIOS的工作就是去初始化各种输入输出，初始化显示器这样的基本的工作。

​ 通过低端的1M内存分布我们知道BIOS是位于顶部的0xf0000-0xfffff着64kb的，而0xffff0-0xfffff这最顶上的16B是BIOS的入口地址，我们当然不能奢求16B能做什么初始化的工作，所以当你看到jmp的时候会觉得很合理。

最后讲一下，实模式下的寻址。在这里我们访问地址是靠段基址和段内偏移地址来访问的。但是寄存器只有16位怎么去访问20位大小的地址呢。嘿嘿，我们把段基址*16，这样就让段基址左移了4位，再加上16位的段内偏移，这样我们就能访问1M内的任何地址。那我们的第一个指令举例：

16 * 0xf000 + 0xfff0 # in hex multiplication by 16 is
= 0xf0000 + 0xfff0 # easy–just append a 0.
= 0xffff0

part2的下面部分都是介绍elf文件格式。

我们的编译器吧各种.c文件编译成.o的重定位文件，这个重定位文件的名字还是很形象的。这个文件里面的各个符号都需要重新定位，而.o文件里面呢是二进制的机器码。当重定位文件经过链接之后就是二进制可执行文件。这个就是我们说的elf格式。

我们知道程序里面有各种section，也就是节。而elf里面会把各种属性相同的section合并在一起称为segment也就是段。所以elf文件他的格式就是，开始是一个elf header，这个header里面有各种信息。然后就是program header，这个header就是segment 的信息。向我们感兴趣的一般有：

  1. .text这个就是代码段，整个程序的指令都会放在这个段里面
  2. .data 程序已经初始化之后的数据段
  3. .rodata这个我还是第一次知道，这个是read-only，只读数据段

链接器在计算内存布局的时候会给为初始化的全局变量保存空间，这个段是.bss段,没有在elf的.bss字段存储内容只存了地址和大小。

我们要知道，当我们把kernel加载进内存之后，loader还要分析elf结构将其解析到新的位置，也就是，内核在内存中有两份。一份是直接的拷贝还有一份是各个segment复制到内存后的程序体。

Part 3: The Kernel

Using virtual memory to work around position dependence

用虚拟内存来解决位置依赖。刚才我们看program header 就能看见内核的segment的VMA（link address）和LMA（load address）是不同的。

操作系统内核一般是喜欢放在虚拟地址的高部分，留下底下的低部分给用户进程。比如说32位下，操作系统一般占据这虚拟地址空间的高1G部分。

很多机器的物理地址并没有0xf0100000这么高，所以我们并不是直接把kernel存放在这里。我们依靠cpu的内存管理硬件（页部件之类的）把物理地址的0x00100000（loader加载kernel到的物理地址）映射到0xf0100000（kernel code 期待运行的link address）【这个就是开始分页模式】。这样一来尽管kernel的虚拟地址很高，留下了大量的空间给用户进程。但是它被加载的物理地址就在我们的BIOS上面，已经不在低1MB里面了，因为0x00100000就是1MB，而内核要往上增长，所以我们的PC机要有几兆的内存。

我们后面会从地往上映射256MB的内存，也就是从0x00000000-0x0fffffff 映射到0xf0000000-0xffffffff。我们现在只映射低端的4MB内存。我们用手写静态初始化的页目录表和页表来完成映射，他们被放在kern/entrypgdir.c。在cr0寄存器的pg位被打开之前，内存引用被视为物理地址（严格来讲是线性地址）。在pg位被打开之后，内存引用就被虚拟内存管理硬件映射为虚拟内存。 entry_pgdir把0xf0000000-0xf0400000 和0x00000000-0x00400000都映射为物理地址0x00000000-0x00400000。任何不再这两个范围内的地址都会造成硬件异常，而异常处理现在还没有。
